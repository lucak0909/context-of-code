<task type="auto">
  <name>Task 1: Start Gunicorn in the foreground</name>
  <files></files>
  <action>
On the VM, in an SSH terminal, navigate to the project root and activate the venv:
```
cd /home/student/context-of-code
source venv/bin/activate
```

Confirm working directory before launching:
```
pwd
```
Expected: `/home/student/context-of-code` — if the output is anything else, fix before proceeding (wrong cwd causes `ModuleNotFoundError: No module named 'web_app'`).

Start Gunicorn with 2 workers, bound to all interfaces on port 5000:
```
gunicorn --bind 0.0.0.0:5000 --workers 2 web_app.app:app
```

This command blocks the terminal. Do not background it (`&`) or kill it — it must stay running for the checkpoint verification in Task 2.

Wait for the startup log to confirm all workers booted. Expected output:
```
[INFO] Starting gunicorn X.X.X
[INFO] Listening at: http://0.0.0.0:5000 (XXXX)
[INFO] Booting worker with pid: XXXX
[INFO] Booting worker with pid: XXXX
```

If Gunicorn exits immediately (non-zero), the cause is almost always one of:
- **Wrong cwd**: `ModuleNotFoundError: No module named 'web_app'` → run `pwd`, must be repo root
- **Venv not active**: `ModuleNotFoundError: No module named 'flask'` → run `source venv/bin/activate` and retry
- **Port in use**: `OSError: [Errno 98] Address already in use` → run `sudo lsof -i :5000` and kill the occupying process

Do NOT use `flask run` — the purpose of this phase is to validate Gunicorn specifically.

    Expected responses that ALL count as SUCCESS:
    - HTTP 405 Method Not Allowed (GET request to a POST-only route — proves the app is routing correctly)
    - HTTP 400 Bad Request (route exists but request is malformed)

    - `curl: (7) Failed to connect to 200.69.13.70 port 5000: Connection refused` → port not open or Gunicorn not running
