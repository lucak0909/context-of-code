    (Replace ~/path/to/your/ssh-key with your actual key path)

  <action>
    Run the following commands on the VM in your SSH session. The venv must be activated first.

    Step 1 — Activate the venv (if not already active):
    ```bash
    cd /home/student/context-of-code
    source venv/bin/activate
    ```

    Step 2 — Validate settings (confirms .env loads and all required keys are present):
    ```bash
    python -c "from common.settings import get_settings; print(get_settings())"
    ```
    Expected success output (values will differ):
    ```
    Settings(db_user='db.xxxxx', db_password='...', db_host='aws-0-eu-west-1.pooler.supabase.com', db_port='5432', db_name='postgres', aggregator_api_url='http://127.0.0.1:5000/api/ingest')
    ```
    If you see `ValueError: Missing required env vars for DB connection: user` (or any key name), the .env key names are wrong. Re-check that keys are lowercase (user=, not DB_USER=). Fix by re-running SCP with a corrected local .env.

    Note: If you edit .env on the VM mid-session, always run a fresh `python -c "..."` command rather than re-importing in a running REPL — get_settings() is cached with @lru_cache(maxsize=1) and will return stale values in the same process.

    Step 3 — Run the Supabase connectivity check:
    ```bash
    python -m common.database.test_connection
    ```
    Expected success output:
    ```
    INFO | Connection successful.
    ```
    Failure scenarios and how to diagnose:
    - `ERROR | Failed to connect.` + authentication error: credentials are wrong — re-check Supabase dashboard
    - `ERROR | Failed to connect.` + connection timeout: university network may block PostgreSQL port 5432. Try switching to Supabase's transaction pooler on port 6543 by updating the `port=` in .env to 6543
    - `ModuleNotFoundError: No module named 'sqlalchemy'`: venv is not activated — run `source venv/bin/activate` first
