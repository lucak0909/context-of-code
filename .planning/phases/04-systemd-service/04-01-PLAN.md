  <action>
SSH into the VM at the project path `/home/student/context-of-code`.

**Step 0 ŌĆö Determine worker count:**
Run `nproc` on the VM. Use the result to set `--workers`:
- `nproc` returns 1 ŌåÆ use `--workers 3`
- `nproc` returns 2 ŌåÆ use `--workers 5`
- `nproc` returns anything else ŌåÆ use formula `(2 * nproc_output) + 1`
- If nproc unavailable for any reason ŌåÆ default to `--workers 2`

**Step 1 ŌĆö Verify the .env format:**
Run `cat /home/student/context-of-code/.env` and confirm all lines are bare `KEY=VALUE` format with no `export` prefix and no shell variable expansion (e.g., `$(command)`). The systemd `EnvironmentFile` directive requires this format. If any line has `export`, it must be removed before using EnvironmentFile.

**Step 2 ŌĆö Write the unit file:**
Create `/etc/systemd/system/context-of-code.service` with the following content (substitute `--workers N` from Step 0):

```ini
[Unit]
Description=Context of Code - Gunicorn aggregator
After=network.target

[Service]
User=student
WorkingDirectory=/home/student/context-of-code
EnvironmentFile=/home/student/context-of-code/.env
ExecStart=/home/student/context-of-code/venv/bin/gunicorn \
    --bind 0.0.0.0:5000 \
    --workers N \
    web_app.app:app
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
```

Use `sudo tee /etc/systemd/system/context-of-code.service` or `sudo nano` ŌĆö whichever is available. Confirm the file was written correctly with `cat /etc/systemd/system/context-of-code.service`.

Key decisions embedded in the unit file:
- `User=student` ŌĆö owns the project files, not root
- `WorkingDirectory` ŌĆö repo root is required for `web_app` module resolution (proven in Phase 2)
- `EnvironmentFile` ŌĆö loads project .env; compatible with bare key=value format
- `ExecStart` uses absolute venv path ŌĆö systemd does not search PATH; relative or system gunicorn would fail
- `Restart=on-failure` ŌĆö restarts on crash, respects intentional `systemctl stop`
- `RestartSec=5` ŌĆö 5-second delay between restart attempts; prevents crash-loop hit limit before diagnosis
- `After=network.target` ŌĆö correct for a server binding to 0.0.0.0; network-online.target is NOT recommended for server software per systemd.io

**Step 3 ŌĆö Reload systemd and activate:**
```bash
sudo systemctl daemon-reload
sudo systemctl enable --now context-of-code
```

`enable --now` combines boot registration and immediate start in one command.

**Step 4 ŌĆö Verify the service is running:**
```bash
sudo systemctl status context-of-code
```

Expected output contains: `Active: active (running)` and `Loaded: loaded (...; enabled; ...)`.

- `Failed to load environment files` ŌĆö .env has `export` prefix or shell expansion; remove those lines

<task type="auto">
  <name>Task 2: Configure UFW to permit port 5000 (SRV-03)</name>
  <files>N/A ŌĆö firewall rule, no file artifact</files>
  <action>
UFW is currently inactive on this VM (confirmed in Phase 2). SRV-03 requires that UFW explicitly permits port 5000 and that `sudo ufw status` lists the rule. To satisfy the literal roadmap success criterion, UFW must be enabled and the rule must be visible.

**CRITICAL ORDERING ŌĆö follow this exactly to avoid SSH lockout:**
The VM uses SSH on non-standard port 2214. If UFW is enabled without allowing port 2214 first, the SSH session is permanently lost.

```bash
# Step 1: Allow SSH on the non-standard port FIRST
sudo ufw allow 2214/tcp

# Step 2: Allow the aggregator port
sudo ufw allow 5000/tcp

# Step 3: Verify BOTH rules are listed before enabling
sudo ufw status
# Must show: 2214/tcp ALLOW and 5000/tcp ALLOW before proceeding

# Step 4: Enable UFW (activates rules, persists across reboots)
sudo ufw enable
# Will prompt "Command may disrupt existing ssh connections. Proceed with operation (y|n)?"
# Type: y
# Note: The session remains active because port 2214 was already allowed in Step 1

# Step 5: Confirm UFW is active and rules are listed
sudo ufw status
```

Expected output of final `sudo ufw status`:
```
Status: active

To                         Action      From
--                         ------      ----
2214/tcp                   ALLOW       Anywhere
5000/tcp                   ALLOW       Anywhere
2214/tcp (v6)              ALLOW       Anywhere (v6)
5000/tcp (v6)              ALLOW       Anywhere (v6)
```

If `ufw` is not installed (unlikely on Ubuntu 22.04): `sudo apt install ufw` then repeat the sequence.
